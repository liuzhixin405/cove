## P0 - 关键优化

### 1. 双模型自动切换（Dual-Model Routing）

**状态**：✅ **已完成（当前会话）**

**问题**：原有 `ModelRouter` 虽然定义了 `complexityClassifier`，但只会返回同一个默认模型，根本没有切换能力。用户只能配置一个模型，手动通过 `/model` 切换。

**方案**：
- `config.json` 支持配置两个模型：
  ```json
  {
    "model": "deepseek-v4-pro",      // 复杂任务模型（高级）
    "model_fast": "deepseek-v4-flash" // 简单任务模型（快速/便宜）
  }
  ```
- `ModelRouter` 根据用户输入自动路由：
  - 含关键词（refactor/architecture/design/debug/optimize等）→ 用高级模型
  - 长消息（>500字符）→ 用高级模型
  - 其他简单任务 → 用快速模型
- `Engine.RunMessageWithStream` 每次用户消息前调用 `Route()` 决策

**涉及文件**：
| 文件 | 改动 |
|------|------|
| `internal/config/config.go` | 新增 `ModelFast` 字段，`applyDefaults` 填充默认值 |
| `internal/api/router.go` | `ModelRouter` 增加 `fastModel`，`complexityClassifier` 真正实现双模型切换 |
| `internal/engine/engine.go` | `Engine` 集成 `modelRouter`，迭代前路由模型 |
| `internal/state/state.go` | `AppState` 新增 `ModelFast` |
| `cli/cove/app_bootstrap.go` | 传递 `ModelFast` 到 engine 和 state |

**使用方式**：
```json
{
  "model": "deepseek-v4-pro",
  "model_fast": "deepseek-v4-flash",
  "provider": { "name": "deepseek" }
}
```

### 2. 模型故障转移（ModelFallback + Provider Failover）

**状态**：✅ **已完成**

**问题**：原引擎直接使用 `e.provider.Chat()` 调用单一提供商，API 故障时无自动切换能力，整个对话卡死。

**方案**：
- 引入 `ModelFallback` 管理多提供商链，自动故障转移
- 支持提供商健康状态追踪：`ProviderOK` → `ProviderDegraded` → `ProviderUnavailable`
- 冷却机制：被限流（429）或临时错误（5xx）后自动冷却 60s，超时自动恢复
- 永久错误（401/403）标记为不可用，需手动恢复

**涉及文件**：
| 文件 | 改动 |
|------|------|
| `internal/api/fallback.go` | 新增 `ModelFallback` 完整实现 |
| `internal/api/provider.go` | 保留 `Provider` 接口定义 |
| `internal/engine/engine.go` | `Engine.provider` 改为 `Engine.fallback *ModelFallback` |

**示例**：
```json
{
  "provider": {
    "name": "deepseek",
    "api_key": "sk-xxx"
  }
}
```
当 DeepSeek 限流时，自动尝试 OpenAI 备用提供商（多 Provider 配置待后续扩展）。

### 3. 三层循环检测（3-Layer Loop Detection）+ 停滞检测

**状态**：✅ **已完成（v6.3.1 升级为 3 层）**

**问题**：Agent 在多轮工具调用中可能陷入重复循环（如反复读取同一文件、反复调用相同工具参数），浪费大量 token 且无实际进展。原简单指纹检测（3次重复即注入提示）过于简单，误报率高。

**方案（实际实现）**：
- **Layer 1a（精确指纹匹配）**：14 轮滑动窗口内同一工具+参数组合出现 ≥10 次则触发
- **Layer 1b（模糊工具名匹配）**：12 轮滑动窗口内相同工具名（不同参数）出现 ≥10 次则触发
- **Layer 2（输出内容哈希）**：40 轮滑动窗口内相同输出哈希出现 ≥8 次则触发
- **Layer 3（停滞检测）**：连续 60 轮无任何文件创建/修改 → 空转检测
- **只读豁免**：read/grep/glob/lsp/webfetch/browser/task_list 不计入循环检测
- **自适应阈值**：Flash 模型使用更敏感的阈值（8/12, 8/10, 8/30, 50）
- **分级响应**：前 5 次非致命注入引导，超出则硬终止

**涉及文件**：
| 文件 | 改动 |
|------|------|
| `internal/engine/loopdetect.go` | 新增 `LoopDetector` 实现 |
| `internal/engine/loopdetect_test.go` | 单元测试 |
| `internal/engine/engine.go` | 集成 `LoopDetector`，替换旧的简单检测 |

### 4. AI 对话压缩（Chat Compression）

**状态**：✅ **已完成**

**问题**：长对话接近 token 上限时，原 `compressHistory()` 仅做简单截断（删除前半段消息），丢失大量上下文信息。

**方案**：
- 当 token 数超过阈值（64K）时触发 AI 压缩
- 使用模型对历史对话进行智能摘要，保留关键决策和结果
- 压缩后的摘要替换原始消息，大幅降低 token 消耗

**涉及文件**：
| 文件 | 改动 |
|------|------|
| `internal/engine/compressor.go` | 新增 `ChatCompressor` 实现 |
| `internal/engine/engine.go` | 集成压缩调用 |

## P1 - 重要优化

### 5. Steer 中途引导机制

**状态**：✅ **已完成**

**问题**：用户在与 Agent 对话时，无法在模型正在执行任务的过程中插入新的指引。需要等待当前轮次完成才能输入新指令。

**方案**：
- 引入 `Steer()` 方法，线程安全地注入用户引导文本
- 引导文本追加到最后一条 tool 消息后，AI 在下一次迭代立即感知
- 支持 `drainPendingSteer()` 在每次迭代前消费待处理引导

**涉及文件**：
| 文件 | 改动 |
|------|------|
| `internal/engine/engine.go` | 新增 `Steer()` / `drainPendingSteer()` 方法 |

**使用方式**：
- REPL 中直接输入内容即可，无需等待当前轮次结束
- 用户引导以 `[用户指引]` 标记出现在 AI 上下文中

### 7. 全面 Hook 系统重构

**状态**：✅ **已完成**

**问题**：原有 Hook 系统只有简单的 `Callback` 函数和 6 个事件，无法满足复杂的生命周期管理需求。

**方案**：
- 重构为 `HookManager` + `HookConfig` 架构
- 支持 11 种事件类型：`BeforeTool`、`AfterTool`、`BeforeAgent`、`AfterAgent`、`BeforeModel`、`AfterModel`、`SessionStart`、`SessionEnd`、`PreCompress`、`PostCompress`、`BeforeToolSelection`、`Notification`
- 支持两种 Hook 类型：Go 运行时回调（`HookRuntime`）和外部命令（`HookCommand`）
- 支持正则匹配器（按工具/模型名过滤）
- 支持同步阻塞和异步触发模式

**涉及文件**：
| 文件 | 改动 |
|------|------|
| `internal/hooks/hooks.go` | 全面重写 |

### 11. 流空闲看门狗（Stream Idle Watchdog）

**状态**：✅ **已完成**

**问题**：流式 API 响应时，如果连接静默断开或服务器停止发送数据，UI 会永久卡在"思考中..."状态。

**方案**：
- 引入 `newStreamWatchdog` 函数，监控流式响应超时
- 180 秒无数据自动取消上下文，触发重试或报错
- 每收到数据块调用 `markProgress()` 重置计时器

**涉及文件**：
| 文件 | 改动 |

| `internal/api/provider.go` | 新增 `newStreamWatchdog()` |
